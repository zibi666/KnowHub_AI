# 项目工作记忆

最后更新：2026-06-09

这个文件用于记录用户在本项目中的长期要求、已踩过的问题、处理经验和后续工作习惯。每次在本项目开始新对话或新任务前，都需要先读取本文件，再结合当前用户的新要求继续更新。

## 每次任务前必须读取

- 先读取 `PROJECT_MEMORY.md`，确认最新项目约定、用户偏好、已知风险和上次结论。
- 再读取 `AGENTS.md`，确认当前仓库级工作规则。
- 如果用户提出新要求、纠正错误、指定长期习惯，任务完成前要把要点整理追加或更新到本文件。

## 用户长期要求

- 项目内文本文件统一按 UTF-8 处理。
- Windows PowerShell 或 cmd 可能以 GBK 显示 UTF-8 中文，出现乱码时不要直接认定文件损坏。
- 遇到“乱码”相关问题，必须用字节级检查、UTF-8 解码或 `repr` 等方式确认真实编码，再决定是否修改。
- 本地构建和运行使用 `.env`。
- 本地调试端口固定为 `8090`，Docker Compose 通过 nginx 暴露 `8090:80`。
- 每次完成任务后，需要重新编译或验证后端和前端，让用户可以在本地调试。
- 这个项目不需要做 UI 层面的浏览器、截图或视觉测试；验证时只做逻辑功能、编译、构建、接口和健康检查。
- 每次完成任务后，需要提交并推送到 GitHub 和 Gitee 的 `design/lm` 分支。
- 不要把临时日志、构建缓存、无关未跟踪文件混入提交。
- 进入聊天页时默认应该停留在新对话空态，不自动恢复上次对话，也不自动打开最后一个历史对话；历史对话只在用户显式点击侧边栏或搜索结果时打开。

## Git 和远端

- 当前主要分支：`design/lm`。
- 推送目标：GitHub 和 Gitee 的 `design/lm`。
- `origin` 配置为一个 fetch URL 和两个 push URL，执行 `git push origin design/lm` 会同时推送两边。

- 工作区可能存在用户未提交改动，必须先看 `git status --short --branch`，只提交本次任务相关文件。

- 后端测试优先使用：`pytest backend\tests -q`。
- 前端构建：在 `frontend` 下运行 `npm run build`。
- 本地 Docker 调试：`docker compose --env-file .env up -d --build`。

## 联网搜索优化方向

- 搜索方案目标是提高准确率、速度和相关性，尤其适配中国服务器环境。
- 不新增付费搜索 API，不引入 Tavily、Firecrawl、Cohere 等新服务，也不做预算控制。
- 总体链路为：用户模式选择、自动分流、快搜/深搜预算、多源并发、中文相关性门控、证据化回答。
- 用户打开联网搜索时，应弹窗选择：
  - 自动
  - 深度优先
  - 快速回答
- 已开启联网搜索后，再次点击图标应打开管理弹窗，允许保存设置或关闭联网搜索，而不是直接关闭。
- 深搜允许用户选择轮数，默认 3，范围 1 到 10。
- 深搜按轮执行：搜索和读取页面，再让模型用 JSON 审查相关性、准确性、证据缺口，并决策是否需要更多关键词或 URL；证据足够时可提前结束。
- 搜索型回答可以展示“搜索过程/证据审查摘要”，只展示检索记录、关键词、读取 URL、证据缺口、停止原因、失败和降级信息，不展示隐藏思考链。
- 中国服务器默认优先 provider：`bocha,sougou,jina,searxng,direct,serper`，缺密钥自动跳过。
- SearXNG 默认引擎以 `bing,baidu` 为主，Google 只作为可选配置。
- 中文相关性采用“宽召回 + 本地明显无关过滤 + AI 语义审查 + 来源证据约束”，不能要求所有中文实体词全字命中。
- 争议、强时效、人物生死、政策、金融、医疗、法律等问题需要更严格的独立来源门槛。

## 已犯过或容易犯的错误

- 不能因为 Windows 终端显示中文乱码，就直接把文件当作损坏编码去批量改写。
- 合并远端更新时，不能只保留本地实现而丢掉远端已经加入的功能。
- 推送失败时不能强推，必须先确认远端新提交并整合。
- 前端临时日志和构建产物不能随手提交。
- 对联网搜索结果不能只看 snippet 就让模型下确定结论，需要正文证据、相关性门控和来源质量分级。
- 给空态输入框左侧工具按钮增加文字标签时，必须同步调整输入框内边距或在窄屏隐藏标签，避免输入文字和工具按钮重叠。
- 空态输入框不要用固定超大左内边距兜所有状态；默认只给左侧图标留紧凑空间，只有联网搜索按钮显示“深搜 10轮”等文字标签时再增加左侧预留，否则新建对话会显得前面空一大截。
- 搜索过程弹窗里 URL、证书错误、失败原因等内容必须强制断行并用自适应布局；不能让长字符串把弹窗横向撑开。
- 深搜可以完整保存 trace 和来源面板数据，但追加给最终回答模型的工具输出和证据上下文必须压缩、限量；否则多轮搜索会把 `/responses` 请求撑大，容易触发上游 `Server disconnected without sending a response`。
- PowerShell 里不要使用 Bash heredoc（`python - <<'PY'`）；需要给 Python 喂脚本时用 PowerShell here-string 再管道到 `python -`。
- 搜索过程弹窗不能只提高弹窗自身 `z-index`；如果它位于 `chat-main/chat-workspace` 的层叠上下文里，还必须在打开时抬高父层级，否则侧栏或底部 composer 仍可能压在弹窗上。
- ORM 删除临时测试用户时，如果 `UserQuota.user_id` 是主键，直接 `db.delete(user)` 可能触发“blank-out primary key”断言；清理临时验证数据时按依赖表顺序显式删除更稳。
- 联网搜索最终回答可引用的来源编号、后端保存的结构化来源上限、前端来源抽屉展示上限必须保持一致；不要让前端用较小硬上限截断来源，否则会出现正文 `[[14]]` 但“已阅读 10 个网页”的错配。
- 来源抽屉和搜索过程弹窗中的 URL、provider 诊断串、失败原因等长文本必须设置 `min-width: 0`、自适应换行和长词断行，避免右侧卡片或弹窗被横向撑开。
- 聊天页启动逻辑不要把 localStorage 里的历史对话 ID 当作默认入口，也不要在无当前对话时 fallback 到最后一个历史对话；否则刷新或重新进入会违背“新对话优先”的预期。
- 修来源抽屉溢出时不能把标题、站点图标、序号和正文摘要挤没；provider、置信度、rerank 等诊断信息应降级为辅助小字，标题和正文摘要必须保持主视觉层级。
- 来源抽屉里的正文摘要不是完整 evidence，也不是抓取脚本/JSON/诊断串；应优先使用干净 snippet 或正文第一句话，超过一两行时以 `...` 结尾，诊断信息只做辅助单行展示。如果候选摘要是代码、JSON、站点导航、播放控件文案、搜索诊断串或几个字的碎片，就不要显示摘要。
- 深搜证据审查超时、失败或不可解析时，不能把 `needs_more` 兜底为 true，也不能塞回原始 query 伪装成继续搜索；应停止补充搜索、用已有证据回答，并在 trace/UI 中显示为已停止。
- 自动联网搜索如果实际策略判定为深搜，就必须执行和显式深搜一致的多轮审查/补充逻辑；不能只把第一轮按 deep 搜索后直接回答。
- 首轮联网搜索 fallback 不能直接拿用户口语化原问题搜索；当模型未主动调用 `search_web` 但联网搜索必须执行时，应先用当前聊天模型以 `low` 思考生成 1 到 3 个首轮关键词，按关键词搜索，规划失败或空结果才回退原问题并写入 trace。
- 深搜证据审查必须把已搜索关键词、已读取 URL、失败 URL 和当前来源覆盖传给审查模型；不要让模型在不知道历史的情况下盲目重复建议关键词。
- 深搜审查模型使用用户当前选择的聊天模型，但审查调用默认 `low` 思考程度，以降低延迟和成本。
- 深搜关键词不要做硬去重；如果审查模型判断本轮某个关键词搜得不精准、噪声大、跑偏、过时或缺少正文级证据，可以继续用相同或相近关键词深挖；如果该关键词已返回精准有用证据，提示词应要求模型不要重复给。代码只应避免同一个失败 URL 反复读取，且不能在审查模型没有给出后续关键词或 URL 时自动把原始问题塞回去伪装成继续搜索。
- 联网搜索最终回答上下文必须优先传正文级 `evidence`，不要优先传 UI 用的短 `snippet`；否则最终回答会因为证据太薄而引用过少。
- 来源摘要 UI 绝不能把 `searxng · 70% · tier:normal · support:high · rerank:fallback` 这类 provider/置信度/诊断串当正文摘要；snippet 不合格时从 evidence 抽正文第一句，仍不合格就不显示摘要。
- 联网搜索最终回答模型只应接收精选 source brief：用户问题、可引用网站 URL、每个网站正文摘要和简短综合回答要求；不要把搜索关键词、失败 URL、审查 JSON 或每轮工具结果日志传给最终回答模型。
- 多轮深搜来源超过 10 个时，最终 prompt 只选全部轮次合并后的 Top 8-10 个高质量独立来源；完整来源和搜索过程保留在 UI trace/source panel。
- 联网搜索最终回答如果上游模型输出 `<tool_call>`、`<tool_call code>` 或 JSON 工具参数，必须在后端清洗掉，不能让伪工具调用作为正文展示给用户。
- 来源抽屉卡片不要展示 provider/confidence/tier/support/rerank 诊断串；这些诊断信息只能留在搜索过程 trace 或调试日志中，不能出现在正文摘要区域。

## 好的方法

- 修改文件前先看 `git status --short --branch`。
- 对乱码问题先做 UTF-8 字节检查，再决定是否修复。
- 对搜索质量问题，把 provider、scraper、reranker、citation/evidence 分层处理。
- 对中文搜索，用可选 `jieba` 分词加 n-gram 兜底，先宽召回，只过滤明显无关、导航页、垃圾页和低质聚合页，再交给 AI 判断语义相关性和准确性。
- 对深搜，不暴露隐藏思考链，只保存结构化检索摘要和 JSON 决策。
- 对 composer 空态布局，优先用 CSS 变量统一管理左右工具区预留空间，修改工具按钮宽度后同时验证输入文本起点。
- 对联网搜索回答链路，区分“给用户审计看的 trace/来源”和“给模型生成看的压缩证据”；后者要有来源数、单条证据长度和总字符数上限。
- 对联网搜索来源 UI，优先保留后端结构化来源里的原始 `index`，让引用按钮、来源抽屉编号和模型生成的 `[[n]]` 一一对应；前端去重或展示截断不能重新编号导致引用错位。
- 对深搜审查链路排查，要先看 `web_search_review_start/done/timeout/failed` 的 `round`、`elapsed_ms`、`payload_chars` 和上游 `responses_first_delta`；如果先收到 HTTP 200/first_delta 后再 timeout，问题是审查总超时切断了未完成 JSON，不是网页抓取失败，审查超时窗口不能短到误杀已开始流式返回的轻量审查。
- 搜索过程摘要要区分配置模式、实际策略、已执行轮次和最大轮数；快速回答不显示最大轮数，只有实际深搜才显示最大轮数。
- 做前端弹窗层级验证时，要看实际截图和 `scrollWidth/clientWidth`、父子层叠上下文、footer/sidebar 是否遮挡，不能只看组件内部没有横向溢出。
- 每次完成后用编译、测试、前端 build、Docker 8090 可用性检查形成闭环。
